# Skill 上下文膨胀 — 5 种行业解决方案对比

**分析时间**: 2026-06-14 22:54  
**适用场景**: WorkBuddy 146 个 Skill 导致的上下文挤占  
**参考来源**: Anthropic 官方、Claude Code 社区、Cursor 社区、MCP 实践者  

---

## 选型标准

对主人的情况，我按**4个维度**打分：
- **效果** — 能省多少上下文（省得越多分越高）
- **成本** — 实现/迁移的工作量（越低分越高）
- **可持续性** — 长期维护的负担（越低分越高）
- **灵活性** — 是否按需启用/禁用，不丢功能

---

## 方案一：Skill 分层暴露（Progressive Disclosure）⭐️⭐️⭐️⭐️⭐️（5星·最优）

### 来源
Anthropic 官方工程博客（2025.9）— 针对 MCP 工具集膨胀的官方解法

### 核心思想
**不是"把所有工具塞进提示词"，而是"让 Agent 自己发现它需要什么"**

Anthropic 团队发现：传统做法把全部 MCP 工具的定义（名称、描述、参数 schema）一次性加载到系统提示词中，这在几十个工具时尚可接受，一旦超过 100 个，上下文就炸了。他们的解法是：**只在系统提示词中保留轻量级的索引（工具名 + 一行描述），完整的工具加载放到运行时按需获取**。

### 在 WorkBuddy 中的落地
```
当前（146个全量加载）:
  系统提示词 = [全部146个SKILL.md的描述] + [记忆] + [规则] + [用户输入]
  → 30K+字符 → 模型推理空间被严重挤占

分层方案（按需加载）:
  系统提示词 = [轻量索引（146个Skill的名称+一行描述）] + [记忆] + [规则]
  → 5K~8K字符 → 模型有余力推理
  → 当AI判断"这个任务需要xxx Skill"时，才去读取完整的SKILL.md
```

### 评分

| 维度 | 分数 | 理由 |
|------|------|------|
| 效果 | ⭐⭐⭐⭐⭐ | 上下文占用减少 70%~85%，最彻底的方案 |
| 成本 | ⭐⭐⭐ | 需要系统层面支持（WorkBuddy 要改 skill 加载机制） |
| 可持续性 | ⭐⭐⭐⭐⭐ | 加新 Skill 不增加上下文负担 |
| 灵活性 | ⭐⭐⭐⭐⭐ | 全部 Skill 可用，只是不预加载 |

### 实施路径
1. Skill 描述精简到 15~30 字（一行能看懂用途就行）
2. 系统提示词中的 `<available_skills>` 只列出名称 + 一行描述（40~60 字符/个）
3. 当 AI 决定调用某个 Skill 时，才完整读取其 `SKILL.md`
4. 自动化任务提示词中明确标记"本次任务需要哪些 Skill"

### 注意
这是一个**系统级改法**，需要 WorkBuddy 平台支持。目前不能由主人的配置层面实现，但可以**向 WorkBuddy 团队提 Feature Request**。

## 方案二：禁用/降级不常用 Skill（skillOverrides）⭐️⭐️⭐️⭐️⭐️（5星·即时可行）

### 来源
Claude Code 社区（2026.5）— 经过验证的 Skill 管理机制

### 核心思想
**Skill 有 4 种状态**，不常用的转到低开销状态即可：

| 状态 | 上下文占用 | AI 能否自动触发 | 用户能否手动调用 |
|------|-----------|----------------|----------------|
| `on`（启用） | 名称+完整描述 | ✅ | ✅ |
| `name-only`（仅名称） | 仅名称 | ✅（但不精准） | ✅ |
| `user-invocable-only`（仅手动） | 不占用 | ❌ | ✅ |
| `off`（关闭） | 完全不占用 | ❌ | ❌ |

### 在 WorkBuddy 中的落地

**这是一个即时可行的方案**，不需要等系统升级。

具体做法：
1. **统计 Skill 使用频率** — 从 `skill-usage-report` 仓库的统计数据中，查出最近 30 天调用次数
2. **分类处理**：
   - **高频 Skill**（如 lobster-comm、lobster-mem0、wechat-clawbot-notify）→ `on`
   - **低频但有用的 Skill**（如 sahaj-yoga-universal、flowsint）→ `name-only` 或 `user-invocable-only`
   - **废弃 Skill**（长期未调用、已合并的）→ `off`
3. **目标**：从 146 个降到 30~50 个活跃 Skill

### 评分

| 维度 | 分数 | 理由 |
|------|------|------|
| 效果 | ⭐⭐⭐⭐ | 能省很多（具体取决于关了多少），但不能从根本上解决 |
| 成本 | ⭐⭐⭐⭐⭐ | 零成本，统计→分析→决策→执行，1~2小时 |
| 可持续性 | ⭐⭐⭐ | 需要定期维护（每月一次清理） |
| 灵活性 | ⭐⭐⭐⭐ | 可以随时开关，不丢文件 |

### 实施路径
1. 用 `skill-usage-report` 看哪些 Skill 最近 30 天零调用
2. 零调用的 Skill 设为 `off`（不是删除，是隐藏）
3. 偶尔用的 Skill 设为 `name-only`
4. WorkBuddy 启动时只加载活跃 Skill → 上下文立竿见影地空出来

## 方案三：上下文压缩（Context Compaction）⭐️⭐️⭐️⭐️（4星·辅助方案）

### 来源
Anthropic 官方 + Claude Code 社区 — `/compact` 命令背后的机制

### 核心思想
**不要等到上下文满了才处理，在 ~70% 使用率时主动压缩**

压缩过程 = 把当前的对话历史喂给模型 → 模型提炼出"关键决策、遗留问题、下一步计划" → 丢弃冗余的工具结果和历史消息 → 在新的会话中继续。

### 在 WorkBuddy 中的落地

自动化任务的 `memory.md` 本身就是一个简易的 compaction 机制：
```
当前：每次自动化都从零开始，读文件→搜mem0→发消息→写日志→复盘
改进：自动化开始时先读自己的 memory.md 获取上次执行的关键结论
      → 跳过重复的初始化步骤 → 节省 token
```

### 评分

| 维度 | 分数 | 理由 |
|------|------|------|
| 效果 | ⭐⭐⭐ | 解决的是"对话历史过长"的问题，不是"Skill描述过多"的问题 |
| 成本 | ⭐⭐⭐⭐ | 可以靠优化 automation memory 实现，无需系统改动 |
| 可持续性 | ⭐⭐⭐⭐ | 一次设置持续生效 |
| 灵活性 | ⭐⭐⭐⭐⭐ | 不影响任何功能 |

### 注意
这个方案**不能替代 Skill 削减**，它是辅助方案。解决的是"对话越长越慢"的问题，不是"一开始就慢"的问题。

## 方案四：Subagent 架构隔离 ⭐️⭐️⭐️⭐️⭐️（5星·已有基建）

### 来源
Anthropic 官方（How we built our multi-agent research system）+ WorkBuddy 已有 `subagent-driven-development` 和 `dispatching-parallel-agents` skill

### 核心思想
**复杂任务不是让一个 Agent 做到底，而是让主 Agent 分解任务、派发给 Subagent、回收摘要**

Subagent 的上下文在自己那里消耗，摘要（~1000-2000 tokens）才回到主 Agent。这样主 Agent 的上下文始终保持干净。

### 在 WorkBuddy 中的落地

**主人已经有了这个能力的大部分基建**：
- `subagent-driven-development` skill ← 直接可用
- `dispatching-parallel-agents` skill ← 直接可用
- `long-task-detach` skill ← 直接可用

**缺的环节**：自动化任务中默认使用 SubAgent 模式

### 评分

| 维度 | 分数 | 理由 |
|------|------|------|
| 效果 | ⭐⭐⭐⭐ | 自动化任务上下文环境改善明显，但不是针对 Skill 描述的 |
| 成本 | ⭐⭐⭐⭐ | Skill 已存在，只需修改自动化 prompt |
| 可持续性 | ⭐⭐⭐⭐ | 一次修改持续生效 |
| 灵活性 | ⭐⭐⭐⭐ | 可按任务类型开关 |

## 方案五：系统 prompt 轻量化 + max_tokens 扩容 ⭐️⭐️⭐️（3星·治标但快）

### 来源
Cursor 社区（Dre Dyson 的上下文优化实践）+ 通用 LLM 最佳实践

### 核心思想
**在不能改系统机制的前提下，从两头下手**：
1. **入口瘦身**：精简 Skill 描述，减少 token 消耗
2. **出口扩容**：增加 max_tokens，给模型更多输出空间

### 在 WorkBuddy 中的落地

```
入口瘦身：
  当前习惯：Skill 描述写 50~200 字，包含用法、触发词、注意事项
  优化后：只保留 15~30 字/个（≈ 一句话说清楚"这 Skill 做什么的"）
  → 146 × (50字→15字) = 省 ~5,000 字符

出口扩容：
  当前：模型可能默认 max_tokens = 4096
  优化后：增加到 8192 或 16384
  → 输出空间扩大一倍，消息截断概率大幅降低
```

### 评分

| 维度 | 分数 | 理由 |
|------|------|------|
| 效果 | ⭐⭐⭐ | 有明显改善但治标不治本 |
| 成本 | ⭐⭐⭐⭐⭐ | 只需改描述文本和配置参数 |
| 可持续性 | ⭐⭐ | 加新 Skill 会重新变胖 |
| 灵活性 | ⭐⭐⭐ | 不影响功能使用 |

| 优先级 | 方案 | 评分 | 对主人的适用性 | 建议立即执行 |
|-------|------|------|--------------|------------|
| **P0** | 方案一（分层暴露） | ⭐⭐⭐⭐⭐ | 需要 WorkBuddy 支持，短期内需提Feature | ✅ 向 WorkBuddy 提需求 |
| **P0** | **方案二（禁用/降级）** | ⭐⭐⭐⭐⭐ | **即时可执行，效果立竿见影** | **✅ 现在就能做** |
| **P1** | 方案五（描述精简+扩容） | ⭐⭐⭐ | 可以同时做，和方案二不冲突 | ✅ 可以并行执行 |

## 最优组合方案（针对主人的情况）

**建议立即执行** = 方案二（禁用/降级 Skill） + 方案五（精简描述+扩容）

不冲突，可以同时做。具体步骤：

### Step 1：统计 Skill 使用频率
从 `skill-usage-report` 或手动扫描 `~/.workbuddy/skills/`，看哪些 Skill 最近 30 天零调用。

### Step 2：分类处理
```
on（保留在活跃列表）     = 最近高频调用的 20~30 个 Skill
name-only（仅名称）      = 偶尔用但还有价值的 Skill
off（完全隐藏）          = 废弃/合并/从不用的 Skill（不是删除！）
```

### Step 3：精简 Skill 描述
把保留的活跃 Skill 描述从 50~200 字精简到 15~30 字。

### Step 4：配置扩容
找 WorkBuddy 的配置或 API 参数，把 max_tokens 调大。

### Step 5：长期策略
向 WorkBuddy 提 Feature Request，请求支持分层加载（方案一）—— 这是根本解法。
