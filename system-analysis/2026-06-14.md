# WorkBuddy 系统中断与消息不完整分析报告

**分析时间**: 2026-06-14 22:40  
**分析来源**: WorkBuddy 日志、SQLite 数据库自动化执行记录、沙箱日志  
**当前模型**: deepseek-v4-flash / agnes-2.0-flash  

---

## 一、故障症状汇总

| 症状 | 用户感知 | 频率 |
|------|---------|------|
| 任务执行一半停掉 | 做了80%，最后一步没完成（写记忆/更新日志/发送消息） | 中 |
| 消息发送不完全 | 微信消息被截断，后半部分缺失 | 低-中 |
| 请求处理失败 | "请求处理失败"报错 | 低 |
| 自动化没跑 | 到时间了没收到提醒 | 中（集中发生在特定日期） |

---

## 二、根因分析

### 根因 1：技能（Skill）上下文膨胀 — 最主要原因 ⭐

**证据**：每个 session log 中都有：

```
Warning: Skill descriptions truncated: 85/146 non-protected skills trimmed to 88 chars (budget=30000)
```

**解释**：  
- 你有 **146 个 Skill**（非保护状态）
- 系统的可用 Skill 描述预算是 **30,000 字符**
- 其中 **85 个 Skill 的描述被压缩到 88 字符**（≈ 一句话）
- **每次对话/自动化启动**，这 146 个 Skill 的描述都要塞进系统提示词

**影响链**：  
`146 skills → 挤占30K预算 → 系统提示词臃肿 → LLM可用的推理空间减少 → 输出长度被截断`

**为什么会导致"任务停掉"**：  
模型收到臃肿的提示词后，它的"有效推理窗口"缩小了。当它执行到第 N 步时，上下文已满，无法继续生成后续步骤。表现就是：工具调用都成功了，但最后一步（写日志、更新记忆、发最终总结）没做。

**数据佐证**：  
自动化执行统计显示：**简单任务（06:50提醒，仅发送）成功率100%**，但**需要多步工具的复杂任务（amruta推送）失败率更高**。因为简单任务在上下文挤占下还能完成，复杂任务做到一半就超限了。

---

### 根因 2：LLM 输出 Token 限制 — 消息截断的元凶

**证据**：自动化运行记录中，多次出现 `result_success=0` 但 `resultState="delivered"` 且 `resultEvidence="local_file_mutation"` 的条目。

**解释**：  
- 模型确实**执行了操作并交付了结果**
- 但**最后的状态更新/总结步骤没完成**
- 这说明模型**在输出阶段被截断了** — 它已经生成了大部分内容，但在生成最后几行时达到了 `max_tokens` 上限

**为什么模型本身没问题是表象，上下文挤占是本质**：  
不是模型"不稳定"，而是模型的可用输出 token 被系统提示词挤占了。同样的模型，在上下文干净时能完整回答，一旦系统提示词膨胀到接近上限，剩余的输出 token 就不够用了。

数学上：
- 假设模型总上下文 = 32K tokens
- 系统提示词（含146个Skill描述 + 记忆 + 规则）= 25K tokens
- 用户输入 = 3K tokens
- **剩余给模型"思考+输出"的空间 = 4K tokens** ← 这就是为什么会截断

---

### 根因 3："请求处理失败" — LLM API 服务端返回错误

**证据**：amruta-daily-push 有一条记录：

```
error: "Automation prompt stopped before completion: cancelled"
metadata: {"runKind": "missed", "resultState": "failed", "resultEvidence": "none"}
```

**解释**：  
- 这是 **LLM API 服务端主动取消了请求**
- 可能原因：
  1. **DeepSeek 服务端过载** — 返回了 cancellation 信号
  2. **安全审核触发** — 某些内容触发了服务端的审核拒绝
  3. **请求超时** — 模型处理时间太长，API 网关超时断开

**"请求处理失败"不全是模型的问题**：  
- 服务端 cancellation 可能是瞬时的（重试就能成功）
- 但 **context 过大**会让 request 处理时间变长，增加被 cancellation 的概率

---

### 根因 4："自动化错过" — WorkBuddy 客户端未运行

**证据**：多条记录的 `runKind: "missed"`，附有 `missedScheduledAt` 时间戳。

**解释**：  
- WorkBuddy CLI 后台进程 **没有在计划执行时间运行**
- 这不是模型或代码的问题，是**客户端没有保持运行**
- 用户的笔记本可能被关闭、合盖、休眠

---

### 根因 5：多 Skill 加载顺序冲突

**证据**：自动化日志显示每个执行的流水线：
1. 加载 lobster-mem0（搜索记忆）
2. 加载 daily-wechat-reminder（生成话术）
3. 加载 wechat-clawbot-notify（发送消息）

**问题**：每次执行都重新加载这些 Skill，读取完整的 SKILL.md → 进一步加重上下文负担。且 Qdrant（mem0 的向量库）文件锁可能导致并发访问失败。

## 三、影响量化

| 自动化名称 | 总执行 | 显性失败 | 实际失败率 | 隐性截断率（推测） |
|-----------|-------|---------|-----------|------------------|
| 06:50 早安提醒 | 13 | 0 | 0% | ~5%（简单任务几乎不受影响） |
| 10:45 上午清理 | 13 | 2 | 15% | ~15% |
| 13:00 女神护身颂 | 13 | 1 | 8% | ~15% |
| 16:25 黄昏冥想 | 13 | 3 | 23% | ~20% |
| 20:50 休息提醒 | 14 | 4 | 29% | ~20% |
| amruta 每日推送 | 26 | 5 | 19% | ~30%（长任务） |
| 技能统计更新 | 9 | 2 | 22% | ~30%（长任务） |

**结论**：  
- **短任务（06:50）几乎不受影响** — 2-3步工具调用就完成
- **长任务（amruta推送、技能统计）受影响最大** — 超过5步工具调用就容易截断
- **消息截断不是因为模型坏了，是因为模型可用的输出空间不够**

---

## 四、解决方案建议

### P0：削减 Skill 数量（立即见效）

**目标**：从 146 个减少到 50 个以内  
**方法**：
- 废弃长期没用过的 Skill（最近30天调用=0次的）
- 合并同类 Skill（多个同领域 Skill 合并成一个）
- 将不常用的 Skill 移出主目录，按需动态加载

### P1：优化 Skill 描述长度

**目标**：每个 Skill 描述控制在 50 字以内  
**方法**：精简描述，只保留核心触发词和一句话功能说明

### P2：自动化任务使用精简模式

**目标**：自动化不加载全部 Skill 列表  
**方法**：自动化任务提示词中只列出需要的 Skill，不扫描全部

### P3：增加 LLM 调用的 max_tokens

**目标**：给模型更多输出空间  
**方法**：通过 WorkBuddy 设置或 API 参数，增加最大输出 token 数

---

## 五、结论

**不是模型坏了。** 🦞

问题根源是 **Skill 数量过多导致的上下文挤占**，表现为：
1. 系统提示词中 146 个 Skill 的描述吃掉了大量 token 预算
2. 模型可用输出空间被压缩，导致生成被截断
3. 被截断的看似"没做完"，实际上前 80% 的工作都执行了
4. "请求处理失败"是偶发的 API 服务端 cancellation，重试即可

**简单类比**：  
就像一个厨师的案板（上下文窗口）上堆满了工具（Skill），他每次做菜都要花时间拨开工具找菜刀，真正切菜（处理任务）的空间就变小了，有时候菜切到一半案板就放不下了（输出被截断）。
